# v3.7-preflight 胜负闭环最小迁移记录

完成日期：2026-07-05

## 1. 性质

本轮是完整 v3.7 发布候选前置补洞，不是完整 v3.7 发布候选。

目标是把 `wude_618_guanzhong_luoyang` JSON 中已经存在的核心胜利目标接入运行时 `VictoryRules`，让默认隋唐剧本具备最小胜负闭环。存档、新手引导、设置、发布候选检查清单、外交归附、州郡经营命令和正式地图资产仍不在本轮范围内。

## 2. 实现范围

### 2.1 MapState objective id 查询

`MapState` 新增：

```text
objective(id:)
controllerOfObjective(id:)
```

运行时胜负判断使用 objective id，例如 `obj_changan`、`obj_luoyang`、`obj_luokou`、`obj_tongguan`，避免依赖中文展示名。

### 2.2 VictoryReason 隋唐原因

`VictoryReason` 新增：

```text
tangControlsLuoyangAndLuokou
luoyangSuiBreaksTongguan
tangHoldsChanganAtFinalTurn
tangLosesChanganAtFinalTurn
```

并新增 `displayName`，供 HUD 展示中文短原因。legacy 阿登原因保留。

### 2.3 VictoryRules 分支

`VictoryRules.updateVictoryState` 现在先按 `scenarioId` 分流：

```text
wude_618_guanzhong_luoyang -> 隋唐胜负分支
其他 -> legacy Ardennes 分支
```

`wude_618` 最小规则：

- 唐控制洛阳和洛口仓，即唐胜。
- 洛阳隋控制潼关，即洛阳隋胜。
- 到 `maxTurns` 后，等待当前势力顺序的最后一个势力结束行动，再按长安控制权结算：
  - 唐控制长安，唐胜。
  - 其他势力控制长安，该控制者胜，原因记为终局长安失守。
  - 长安没有控制者时不强行写 winner。

终局检查等待最后一个势力，是为了避免在 final turn 第一个势力刚结束时过早结算。

### 2.4 HUD 胜负显示

HUD 的“胜负”字段现在显示：

```text
未定
唐胜：克洛阳与洛口仓
洛阳隋胜：夺潼关入关中
```

实际展示取决于 `VictoryState.winner` 和 `VictoryState.reason`。

## 4. 架构边界

- 本轮不改 `Command`、`ZoneDirective`、`WarCommandExecutor`、`RuleEngine` 执行管线。
- 胜负只读取 `HexTile.controller` 派生出来的目标点控制者，不直接修改 hex、region、theater、front 或 deploy。
- 不改 `regionToTheater`、`hexToTheater`、`hexToFrontZone` 权威边界。
- 不把 `VictoryConditionDefinition` 扩成通用解释器，避免在发布候选前制造大 schema 迁移风险。
- `RegionVictoryRules` 保持 legacy 参考，不作为 `wude_618` 主路径。
