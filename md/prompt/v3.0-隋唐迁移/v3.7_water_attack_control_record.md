# v3.7-preflight.112 隔河近战水路控制校验记录

切片 ID：v3.7-preflight.112

来源：`codex-v3.0-隋末唐初aiagent历史策略迁移总提示词.md` §0.2 当前队列中的“完整水战与更细水域控制规则”，并响应人工最新方向：隋唐古代作战应重视渡口、粮道、骑兵、关隘和天下局势，不应继续按现代连续战线感觉推进。

当前基线：

- v3.7-preflight.103 已让渡口/港口/海港影响粮道跨河补给成本。
- v3.7-preflight.106 已让己控渡口/港口/海港成为战术补给投送点。
- v3.7-preflight.108 已让己控渡口/港口/海港减免相邻跨河移动成本。
- v3.7-preflight.109/.110 已让己控港口/海港进入生产部署和 region 战略补给源。
- 战斗校验此前只看单位、外交和射程；普通近战可无条件隔河攻击。

## 目标

- 对相邻、隔 `riverEdges`、且攻击方没有远程支援的攻击增加水路控制门禁。
- 攻击方控制攻击端或防守端的渡口、港口或海港时，隔河近战可继续执行。
- 缺少己控水路通行点时，返回明确的 `waterCrossingBlocked` 校验错误。

## 非目标

- 不新增水战命令、水域控制模型、运输/登船、船队状态或水域 ZOC。
- 不改变移动、补给、经济、AI directive、JSON schema 或 MapEditor 行为。
- 不把水师部署、港口募兵、河面封锁混进本轮。

## 实现摘要

- `CommandValidationError` 增加 `waterCrossingBlocked`，显示“缺少己控水路通行点”。
- `CommandValidator.validateAttack` 在射程和外交校验之后调用隔河近战门禁。
- 只有同时满足以下条件才触发门禁：
  - 攻击者与目标相邻。
  - 两格之间存在 `riverEdges`。
  - 攻击者没有 `hasRangedSupport`。
- `hasControlledWaterTransit` 复用与移动/补给一致的水路点口径：marker kind 为 `ferry` / `port` / `harbor`，且对应 hex `controller == attacker.faction`。
- `RuleEngineCoreTests` 增加无己控渡口拒绝、己控渡口允许、水师跨河攻击允许的语义断言。

## 人工方向落实

- 本轮不是完整水战，而是把古代渡口/河道控制先接入战斗入口，避免普通陆军像现代机械化战线一样无条件跨河冲击。
- 下一步应优先围绕古代作战重塑继续切片：
  - 弱化玩家可见“前线/战线”术语，改成边境接触、战场态势、行军道威胁。
  - 强化骑兵机动、突击、追击和名将统率。
  - 强化粮草、粮仓、府库、补给压力对行动和 AI 目标的影响。
  - 强化隋唐英雄人物在首屏、战报、军令和外交中的存在感。

## 风险与后续

- 本轮没有启动 App，因此玩家攻击反馈和 UI 布局仍需运行时复核。
- `CommandValidator` 与 `MovementRules` 暂时各自拥有 `hasControlledWaterTransit` 近似 helper；后续若继续扩展水路控制，应抽到共享规则层，避免口径分叉。
- 完整水战、水师部署、河面控制、运输/登船和水域 ZOC 仍需独立设计。
