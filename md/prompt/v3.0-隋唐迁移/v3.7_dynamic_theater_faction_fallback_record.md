# v3.7-preflight.97 动态方面推进势力兜底收口记录

完成日期：2026-07-06

## 目标

继续处理总提示词 §0.2 的 P1 风险：`WarCommandExecutor.applyStrategicAdvance` 在动态方面推进时，如果 `advancingZoneId` 对应的 front zone 缺失，会把 `TheaterSystem.expandDynamicTheater` 的推进势力静默兜底为 `.germany`。这会让异常隋唐或自定义场景路径把动态方面正式化比例算回旧东路势力。

本轮只收口 `WarCommandExecutor` 的推进势力推断，不改变 hex 占领权威、`hexToTheater` 单 hex 推进规则、`hexToFrontZone` 部署权威、同步器主逻辑、命令 schema、AI 决策或存档格式。

## 改动范围

- `WWIIHexV0/Commands/WarCommandExecutor.swift`
  - 新增 `resolvedAdvancingFaction(for:advancingZoneId:state:)`，优先读取 `frontZones[advancingZoneId].faction`，异常缺失时回退到实际行动军队的 `division.faction`。
  - 两者都无法确认时写入中文战报“动态方面推进跳过：无法确认推进势力。”并跳过本次动态方面推进。
  - `shouldAdvanceDynamicTheater` 改为接收已确认的 `advancingFaction`，不再因 source zone 缺失而需要 legacy faction fallback。
  - `TheaterSystem.expandDynamicTheater` 不再使用 `.germany` 作为推进势力兜底。

## 明确不改

- 不改 `TheaterSystem`、`WarDeploymentManager`、`StrategicStateSynchronizer` 或 `CommandExecutor`。
- 不改 `HexTile.controller`、`Division.coord`、`regionToTheater`、`hexToTheater`、`hexToFrontZone` 的权威边界。
- 不改变 directive move 成功后的占领、region 同步、front/deploy 刷新顺序。
- 不改 `Command` / `ZoneDirective` / `RuleEngine` / `CommandValidator` schema 或规则数值。
- 不处理 P0 `RegionDataSet`、P2 场景语义 helper 或 P3 MapEditor unit faction fallback。
- 不跑本机 Xcode build / XCTest / 模拟器 / Probe / Smoke / Full。

## 并发子 Agent 审计

本轮主 Agent 启动两个只读子 Agent：一个复核总提示词后续切片队列，一个扫描源码 legacy fallback 残留。子 Agent 不写文件；主 Agent 根据扫描结果选择 P1 作为追加小切片，并负责代码与文档整合。
