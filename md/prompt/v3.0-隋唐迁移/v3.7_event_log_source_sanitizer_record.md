# v3.7-preflight.78 GameLogEntry 源头战报 legacy 文案收口记录

## 目标

收口 Rules / Commands / Data / Core 等源头写入 `GameState.eventLog` 时的 legacy 势力名、旧地名、旧单位词和 raw id 展示风险，避免外部 `gameState.eventLog` 绕过 `AppContainer` 交互日志净化。

## 范围

本轮只改 `GameLogEntry` 消息初始化入口和文档；不改 `GameState.eventLog` 字段结构、`GameLogEntry` schema、`relatedRecordId` 合同、规则数值、命令执行、`Command` / `ZoneDirective`、`Division.name`、`RegionNode.name`、`FrontZone.name`、JSON、id / rawValue、AI 决策或存档格式。

## 实现摘要

- `GameLogEntry.init(...)` 对传入 `message` 做源头展示净化后再保存到 `message`。
- 净化覆盖常见审计 id、region / theater / front zone / objective / hex / division / unit / command / agent raw id。
- 净化覆盖 legacy 势力显示“德军（旧）/ 盟军（旧）/ 德军 / 盟军”。
- 净化覆盖旧地名和旧题材单位词：`阿登`、`巴斯托涅`、`圣维特`、`Ardennes`、`Bastogne`、`St. Vith`、`St Vith`、`Sedan`、`装甲`、`摩托化`、`炮兵`、`步兵`、`反装甲`。
- 既有存档或数据加载中直接构造的 `GameLogEntry` 也会在解码 / 初始化时进入同一净化入口。

## 并发扫描结论

- Rules 扫描指出 `CommandExecutor`、`SupplyRules`、`EconomyRules` 多处 `appendEvent` 直接拼接 `division.name`、`region.name`、`faction.displayName`、`record.summary`。
- Commands 扫描指出 `WarCommandExecutor` 与 `StrategicStateSynchronizer` 的控制权 / 推进战报直接拼接 region、faction 和 theater 展示名。
- 两组扫描均建议不要改变规则行为或 record id 合同；本轮采用 `GameLogEntry` 统一入口净化，覆盖所有 `appendEvent` 调用和直接 `GameLogEntry(...)` 构造。

## 边界

- 不修改任何规则层判断、资源扣减、战斗伤害、补给、撤退、经济、外交或动态方面 / 防区推进语义。
- 不修改 `relatedRecordId` 存储；展示层仍可继续净化 record id。
- 不修改 `Faction.displayName`、`Division.name`、`RegionNode.name`、`FrontZone.name` 或 JSON 数据。
- 不全局替换单字“师”，避免误伤“军师”等普通文本；只替换明确旧兵种词。
