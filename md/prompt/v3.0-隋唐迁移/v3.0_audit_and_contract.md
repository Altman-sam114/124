### 3.2 二元阵营与 `.opponent` 假设

命令：

```sh

```

命中重点：

| 文件 | 风险 |
|---|---|
| `WWIIHexV0/Core/Faction.swift` | `Faction` 只有 `germany/allies`，`opponent` 是二元翻转 |
| `WWIIHexV0/Rules/SupplyRules.swift` | 补给点、可占领地块判断仍用 `faction.opponent` |
| `WWIIHexV0/Rules/RegionSupplyRules.swift` | region 补给通行和安全撤退仍用 `faction.opponent` |
| `WWIIHexV0/Rules/RegionCombatRules.swift` | 区域压力只扫描 `faction.opponent` |
| `WWIIHexV0/Rules/FrontLineManager.swift` | fallback front line `factionB` 用 `factionA.opponent` |
| `WWIIHexV0/Commands/WarCommandExecutor.swift` | 战术目的地排序偏好 `division.faction.opponent` 控制 hex |
| `WWIIHexV0/Agents/AgentContexts.swift` | 旧 Agent D 摘要用 `agent.faction.opponent` 找敌军 |
| `WWIIHexV0/Agents/ZoneCommanderAgent.swift` | 目标丢失统计用 `faction.opponent` |

判断：

- `.opponent` 是 v3.1 最大阻塞点。多势力下敌我必须来自 `DiplomacyState` / 关系服务，不能继续二元翻转。
- `.opponent` 可短期保留给旧测试或 legacy 兼容，但 v3.1 新主链路不得新增依赖。

## 4. v3.0 迁移合同

### 4.1 状态权威合同

后续所有隋唐迁移仍必须遵守：

- `HexTile.controller` 是真实占领权威。
- `Division.coord` 是单位位置权威，短期可保留源码名 `Division`。
- `RegionNode.controller` 只能由 region 内 hex controller 聚合，不得反向覆盖 hex。
- `regionToTheater` 只作为初始/基础方面归属和地图编辑器种子。
- `hexToTheater` 是运行时动态方面权威；突破只推进单个 hex。
- `hexToFrontZone` 是部署层动态归属权威；`regionToFrontZone` 只能作 dominant / fallback。

### 4.2 命令入口合同

玩家、AI、聊天命令、MockAI、后续君主/谋主/总管/太守/使者 Agent 都必须收口到：

```text
Command / ZoneDirective
  -> WarCommandExecutor
  -> CommandValidator
  -> RuleEngine
  -> CommandExecutor
  -> StrategicStateSynchronizer
```

禁止：

- UI 直接写 `GameState`。
- Agent 直接写 `HexTile.controller`、`Division.coord`、`hexToTheater`、`hexToFrontZone`。
- 外交/归附事件直接把整块 region 改成某势力控制。
- MapEditor 的初始 theater assignment 被运行时当作推进脚本。

### 4.3 势力与外交合同

v3.1 推荐短期方案：

- `Faction` 保留为 Codable 兼容入口，但扩展为隋唐主要势力。
- 建议 case：
  - `tang`
  - `luoyangSui`
  - `wagang`
  - `xia`
  - `qinXue`
  - `liuWuzhou`
  - `tujue`
- 中立继续优先使用 `HexTile.controller == nil` 表达；是否新增 `neutral` case 需谨慎，因为它会影响 `Faction.allCases`、经济账本、外交初始值和 MapEditor picker。
- `Faction.opponent` 标记为 legacy convenience，不再作为新逻辑入口。
- 新增或强化关系判断接口，例如：

```text
DiplomacyState.relationStatus(between:and:)
DiplomacyState.isHostile(_ lhs: Faction, _ rhs: Faction)
DiplomacyState.canAttack(_ lhs: Faction, _ rhs: Faction)
```

约束：

- `target.faction != attacker.faction` 只能作为最小防己方攻击检查，不能代替战争关系。
- 补给、ZOC、前线、AI 摘要、战术目标排序都要使用关系判断。
- 旧 `germany/allies` 存档或测试可通过兼容映射继续读取，但主 UI 不应显示为德军/盟军。

### 4.4 回合阶段合同

当前 `GamePhase.germanAI/alliedPlayer` 不支持多势力。

v3.1 推荐方向：

- 短期新增通用阶段：
  - `playerCommand`
  - `aiCommand`
  - `resolution`
- 保留旧 case 作为 Codable 兼容或迁移适配。
- 命令合法性应基于：

```text
state.activeFaction == command issuer faction
phase allows command
issuer is human controlled / AI controlled / observer mode allowed
```

而不是：

```text
germanAI -> germany
alliedPlayer -> allies
```

### 4.5 数据 schema 合同

后续隋唐 JSON 应使用 ASCII id，中文只放展示字段。

建议命名：

- 势力：`power_tang`、`power_luoyang_sui`、`power_wagang`
- 地区：`region_changan`、`region_luoyang`、`region_hulao`
- 战区/方面：`zone_guanzhong_east`、`theater_hedong`
- 将领：`general_li_shimin`、`general_li_mi`
- Agent：`sovereign_li_yuan`、`strategist_liu_wenjing`
- 剧本：`wude_618_guanzhong_luoyang`

v3.2 之前不得把新字段塞进无关旧字段。schema 暂不支持的城池、粮仓、渡口、关隘、民心、天命等，应先记录在阶段文档或新增明确字段。

### 4.6 单位和经济显示合同

短期源码兼容名与玩家显示名分离：

| 源码/旧语义 | v3.x 显示语义 |
|---|---|
| `Division` | 军队 / 部曲 / 行军队 |
| `tank` | 骑兵 / 突骑，具体需按 v3.3 兵种方案确认 |
| `motorizedInfantry` | 轻骑 / 机动步骑，具体需按 v3.3 确认 |
| `infantry` | 步卒 |
| `artillery` | 弓弩 / 攻城器械，需拆分时另行实现 |
| `manpower` | 丁口 |
| `industry` | 钱帛 / 军械 |
| `supplies` | 粮草 |
| `Panzer Division` | 骑军 / 精骑 / 玄甲军方向，需人工确认 |
| `Supply Stockpile` | 粮草转运 / 粮仓补给 |

v3.1-v3.2 可先做显示映射；v3.3 再改规则权重和兵种差异。

### 4.7 AI Agent 合同

后续隋唐 Agent 分层必须遵守：

```text
SovereignAgent / StrategistAgent / GovernorAgent / DiplomatAgent
  -> Codable directive envelope
  -> decoder / validator / compiler
  -> ZoneDirective / Command
  -> WarCommandExecutor / RuleEngine
  -> WarDirectiveRecord / AgentDecisionRecord / diplomacy record
```

要求：

- 所有 Agent 输出必须 Codable。
- 外部模型输出必须被 decoder 校验。
- schemaVersion、turn、issuerId、power/faction、zone、region、tactic 必须可校验。
- JSON 解析失败时走 deterministic fallback。
- 不提交 API key、个人模型路径或网络端点。

## 7. 风险清单

- 多势力迁移会影响 `Faction.allCases`，进而影响经济账本、外交初始化、MapEditor picker、UI 颜色和测试 fixture。
- 如果直接删除 `germany/allies`，旧 JSON、测试、历史 prompt 和默认数据会大面积失效。
- 如果新增 `neutral` 到 `Faction`，需要防止经济和 AI 把中立当成可行动势力。
- 如果先改 UI 文案但不改规则，可能产生“看起来是隋唐，底层仍按二元战争执行”的误导。
- 如果先改默认剧本但不改 `CommandExecutor` 二元回合轮换，第三方势力无法正常行动。
- 如果绕过 `RuleEngine` 做归附/占领事件，会破坏玩家和 AI 对称性。
- 如果把 `regionToTheater` 当运行时方面推进，会重现动态战区前线跳变问题。
