# v3.7-preflight.23 AI 善后未处置优先治理

## 目标

在 v3.7-preflight.21/.22 已有善后处置审计和进度摘要的基础上，把处置记录反馈给 AI 太守候选排序：最新善后压力记录中尚未产生处置记录的州郡，应排在已处置但仍可治理的州郡之前。

## 范围

- 在 `DiplomacyState` 中新增已处置 / 未处置善后州郡的只读 helper。
- 让 `governedAftermathRegionCount` 复用已处置州郡 helper，避免 UI 进度和 AI 排序口径分叉。
- 调整 `TurnManager.aftermathGovernorRegionIds`：先返回未处置受影响州郡，再返回已处置但仍可治理的受影响州郡。
- 保持 AI 太守每回合最多一条 `Command.governRegion`。
- 更新发布前检查面板、README、核心流程文档、流程图、计划文档和更新日志。

## 非目标

- 不新增命令类型，不增加 AI 回合行动次数。
- 不改变 `Command.governRegion` 的校验、消耗或效果。
- 不清零、降级、删除或解决 `SubmissionAftermathRecord`。
- 不新增忠诚、叛乱、贡赋、俘虏、安置、治安长期状态机或额外资源变化。
- 不批量处理历史善后压力；当前仍只读取最新一条善后记录。

## 数据流

```text
latest SubmissionAftermathRecord
  -> affectedRegionIds
  -> linked SubmissionAftermathGovernanceRecord
  -> DiplomacyState 区分未处置 / 已处置 region ids
  -> TurnManager.aftermathGovernorRegionIds
  -> 未处置州郡优先进入 governorCommand 候选
  -> Command.governRegion
  -> RuleEngine
```

## 架构边界

- Hex 仍是战术权威；本轮不修改 `HexTile.controller`、`Division.coord`、`hexToTheater` 或 `hexToFrontZone`。
- Region 仍是战略聚合层；AI 排序只决定哪个州郡先提交既有经营命令。
- 善后处置记录仍是审计层，不代表善后压力被规则层解决。
- AI 善后治理仍沿用每回合最多一条 `Command.governRegion` 的既有节奏。

## 风险与后续

- 没有真实运行时验证时，AI 多回合是否按预期轮转处理多个善后州郡仍有未确认风险。
- 当前只读取最新善后记录；历史善后压力筛选、分批处理和长期善后状态机需后续单独设计。
- 后续若实现忠诚、叛乱、贡赋、俘虏或安置，必须新增清晰的命令、validator 和 executor，不能让排序 helper 承担规则效果。
